# Task: Refactor LooperExecutor to use @JavaClass / @JavaStaticMethod macros

## What to change

`AppCore/Sources/AppCoreAndroid/LooperExecutor.swift` currently calls two static
methods on the Kotlin `LooperPoster` object using raw JNI — manual `FindClass`,
`GetStaticMethodID`, `NewGlobalRef`, `CallStaticVoidMethodA`,
`CallStaticBooleanMethodA`, string-typed JNI descriptor literals (`"(J)V"`,
`"()Z"`), and a `Mutex<_PosterCache>` to cache the handles across calls.

This is ~60 lines of error-prone plumbing. The `@JavaClass` / `@JavaStaticMethod`
macros from `SwiftJava` (already a dependency) replace all of it with a 5-line
struct declaration. The method signatures are derived from Swift types at compile
time; descriptor strings disappear entirely.

## Proposed replacement

### 1. Declare the Kotlin class as a Swift wrapper

Add (inside `#if canImport(Android)`) — either at the top of
`LooperExecutor.swift` or in a new `LooperPoster.swift` file in
`AppCoreAndroid/`:

```swift
#if canImport(Android)
import SwiftJava

@JavaClass("com.example.appcore.bridge.LooperPoster")
struct LooperPoster: AnyJavaObject {
    @JavaStaticMethod
    func postToMain(_ jobPointer: Int64) {}

    @JavaStaticMethod
    func isOnMainLooper() -> Bool { false }
}
#endif
```

`@JavaClass` generates `fullJavaClassName`, `javaHolder`, `AnyJavaObject`
conformance. `@JavaStaticMethod` generates a body that calls
`dynamicJavaStaticMethodCall(methodName:arguments:resultType:)`, which handles
`FindClass` / `GetStaticMethodID` / caching / dispatch generically.

### 2. Simplify `enqueue` and `checkIsolated`

```swift
func enqueue(_ job: consuming ExecutorJob) {
    let jobPointer = unsafeBitCast(UnownedJob(job), to: Int64.self)
    let env = try! JavaVirtualMachine.shared().environment()
    try! LooperPoster.postToMain(jobPointer, environment: env)
}

func checkIsolated() {
    let env = try! JavaVirtualMachine.shared().environment()
    let onMain = try! LooperPoster.isOnMainLooper(environment: env)
    precondition(
        onMain,
        "LooperExecutor.checkIsolated: not on Android's main Looper. " +
        "JNI thunks may only be called from the UI thread."
    )
}
```

### 3. Delete the now-unused plumbing

Remove entirely:
- `_PosterCache` enum (~10 lines)
- `CachedHandles` struct (~5 lines)
- `lookupCached(env:)` method (~35 lines) including the mutex lock, `FindClass`,
  `NewGlobalRef`, `GetStaticMethodID` calls, and bit-pattern dance

The `cache: Mutex<_PosterCache>` stored property on `LooperExecutor` also goes.

## What does NOT need changing

- `@_cdecl("Java_com_example_appcore_bridge_LooperPoster_runSwiftJob")` — the
  inbound callback that runs a Swift job posted from Kotlin. One function, already
  clean. Leave it alone unless `@JavaImplementation` proves clearly better after
  checking the macro output.

- `JavaInterop.swift` — the five `@unchecked Sendable` extensions on
  jextract-generated sink wrappers. This is a known tooling gap in jextract (it
  doesn't emit `Sendable`). Macros don't fix it; keep as-is.

- Everything in `AndroidSnapshot`, `AndroidBinding`, `AndroidCommands`, `Bridge` —
  not touched by this refactor.

## Why

| Before | After |
|--------|-------|
| Stringly-typed JNI descriptors `"(J)V"`, `"()Z"` | Type-safe — derived from Swift signature |
| Manual `FindClass` + `GetStaticMethodID` + caching | Generated by macro |
| `_PosterCache` mutex + bit-pattern Sendable workaround | Gone |
| ~60 lines of plumbing | ~10 lines total |
| Runtime crash if Kotlin changes signature | Compile-time mismatch (name/type) |
