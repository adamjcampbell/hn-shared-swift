# ADR-0011: Value-carrying typed `*OnChange` callbacks

## Status

Superseded by [ADR-0013](0013-skipfuse-bridgemembers.md) on 2026-05-10. SkipFuse routes `@Observable` notifications into Compose's `MutableStateBacking` snapshot system directly; there is no Kotlin-side callback protocol to deliver values into anymore.

## Context

The hand-written observation pipeline ([ADR-0009](0009-observations-asyncsequence.md)) called a Kotlin callback on every emission. The callback originally fired *just a notification* — `onChange()` with no payload — and the Kotlin side then called a separate `appcoreReadX()` thunk to fetch the new value. Every emission paid two JNI round-trips: one inbound notification, one outbound read.

The pattern had a generic shape on paper — `protocol OnChange { func onChange() }` plus per-property `read` thunks — but in practice every emission carried exactly one new value through the bridge, and that value was always known at the moment the notification fired (the `for await value in Observations(...)` loop body had it in hand). The second JNI round-trip was redundant.

`swift-java jextract --mode=jni` bridges Swift protocols to Java SAM interfaces, but it does not bridge generic protocols: `protocol OnChange<T> { func onChange(value: T) }` would not survive JNI. The way to carry the value through the callback was to define one non-generic protocol per observed Swift type.

## Decision

Replace the bare `OnChange { onChange() }` protocol and the per-property `appcoreReadX()` thunks with typed value-carrying protocols, one per observed type:

```swift
public protocol BoolOnChange: Sendable    { func onChange(value: Bool) }
public protocol StringOnChange: Sendable  { func onChange(value: String) }
public protocol OptionalStringOnChange: Sendable { func onChange(value: String?) }
public protocol LongOnChange: Sendable    { func onChange(value: Int64) }
```

The bridge's `observe<T>` helper picks the right protocol per call site; the Kotlin `OnChange` lambda receives the new value directly in its parameter list. The `appcoreReadX()` thunks are deleted.

## Consequences

- One JNI round-trip per emission instead of two. For a binding emitting at any reasonable rate this is a real cost reduction; for the bindings in this app (search query at user-typing rate, isLoading at fetch boundaries) it was small in absolute terms but eliminated all the boilerplate around the redundant read.
- The bridge gains a typed protocol per observed Swift type — currently `BoolOnChange`, `StringOnChange`, `OptionalStringOnChange`, `LongOnChange`. Each new observed value type adds one protocol. The cost is bounded by the set of observed types, not the set of observed properties.
- Kotlin call sites are slightly cleaner: `OnChange { state.value = it }` reads naturally, with no separate read call surrounded by an explanatory comment.
- The Swift side's `observe<T>` helper signature stays generic — `T: Sendable` — and the per-type protocol selection happens at the JNI thunk's `@_cdecl` declaration. The internal generic shape is preserved.

When SkipFuse replaced the bridge, the per-property accessors generated by `// SKIP @bridgeMembers` deliver value reads through Compose's snapshot system without a Kotlin-defined callback at all — there is no `OnChange` analogue, the snapshot cell *is* the channel. The typed-OnChange protocols, like the rest of the hand-written observation surface, were deleted.
