# Swift-on-Android Architecture Comparison
### observable-cross vs. Skip Fuse vs. the Official Apple/Swift Stack

*Generated 2026-05-08*

---

## Executive Summary

This project (`observable-cross`) and Skip Fuse are both trying to solve the same hard problem — running Swift business logic natively on Android and wiring it to Jetpack Compose — but they occupy different positions in the solution space and make meaningfully different bets.

**observable-cross** is an alignment play: build directly on Apple's official Swift SDK for Android and the swiftlang/swift-java toolchain, contribute upstream rather than wrapping, and keep Android UI as idiomatic Kotlin/Compose. It sacrifices UI code-sharing for architectural clarity and long-term ecosystem health.

**Skip Fuse** is a product play: layer polished tooling and automation on top of the same compiler infrastructure to achieve full SwiftUI-on-both-platforms today, at the cost of a commercial dependency and some bridging opacity.

Neither is wrong. They reflect different priorities.

The verdict: **this project's architecture is sound and its bets are defensible.** There are a handful of concrete techniques from Skip Fuse worth adopting, and a clear case for contributing the project's novel patterns (JavaUIActor, echo-dedup bindings) upstream to swift-java.

## The Landscape

### 1. Apple's Swift SDK for Android (the compiler target)

Stable since Swift 6.3 (March 2026), this is the shared foundation that all native approaches build on. It cross-compiles Swift to ARM64/x86-64 `.so` shared libraries via LLVM, targeting Android's Linux kernel layer — not the JVM. The SDK itself provides no bridge, no UI layer, and no Jetpack Compose integration. It is a compiler target, not a framework.

### 2. swift-java / jextract (the official binding generator)

`swiftlang/swift-java` is the official Swift project's automated JNI codegen tool. Given Swift source files, it generates Java wrapper classes with `static native` methods backed by `@_cdecl`-annotated Swift thunks. JNI mode (the Android-compatible path, landed via GSoC 2025) supports Swift classes, structs, enums, and protocols as Java interfaces. It is the layer that makes the Swift Android SDK practical without hand-writing JNI — and it is the primary tool this project uses.

**Maturity:** Experimental but fast-moving. WWDC 2025 gave it official attention. Breaking changes are expected. The JNI mode's protocol-as-Java-interface feature is notably ahead of where the official docs describe it.

### 3. Skip Lite (transpilation)

The original and most mature Skip mode. A build plugin transpiles Swift source to idiomatic Kotlin before the Android Gradle build runs. No JNI. No native libraries. SwiftUI maps to Jetpack Compose via SkipUI (a full Kotlin reimplementation). The tradeoff: you get JVM semantics (GC replaces ARC, generics are erased), transpiler gaps for advanced Swift, and a dependency on Skip's reimplementation of Foundation/SwiftUI rather than Apple's own. Not relevant to this project.

### 4. Skip Fuse (native Swift + automated JNI bridge)

The newer Skip mode and the closest competitor to observable-cross's approach. Swift compiles natively to `.so` via the same Apple SDK. The JNI glue is auto-generated by the `skip-bridge` build plugin (opt-in via `bridging: true` in `skip.yml`). SwiftUI is translated to Compose via SkipFuseUI → SkipUI. Skip Fuse is free for indie developers as of November 2025; commercial licensing applies to enterprise use.

## Architecture Deep-Dive

### How observable-cross Works

The project's architecture is cleanly split across four layers:

**AppCore (cross-platform):** A `@Observable final class AppState` holds all application state. `AppModel` owns an `AppState` instance and exposes a single `dispatch(_ event: AppEvent) async` entry point. `AppEvent` is a `Codable` enum serialized as JSON for cross-boundary transport. The same `AppModel` runs on both platforms — on iOS with `@MainActor` isolation, on Android with `@JavaUIActor` isolation.

**AppCoreAndroid (JNI bridge layer):** Three reusable primitives, all `@JavaUIActor`-isolated:

- `AndroidSnapshot<S: Encodable>` — Drives an `Observations { … }` loop (SE-0475), encodes the entire state to JSON on each transaction, deduplicates byte-identical snapshots, and fires `SnapshotSink.deliver(snapshotJSON:)` via JNI.
- `AndroidBinding<Root, Value>` — A per-property two-way bridge. Observes a keypath via `Observations`, suppresses echo (writes from Kotlin arrive via JNI setter → `lastSetterValue`; if the observer fires the same value, it skips the JNI callback). Used for `searchQuery` (two-way) and `isLoading` (one-way Swift→Kotlin).
- `AndroidCommands<C: Encodable>` — Drains `AsyncStream<AppCommand>`, encodes to JSON, delivers via `CommandSink`.

**JavaUIActor / LooperExecutor:** A custom SE-0392 global actor whose serial executor posts work items to Android's main `Looper` via JNI (`Handler.post`). This pins all Swift actor isolation to the Android UI thread — JNI thunks call `JavaUIActor.assumeIsolated { }` synchronously, and async Swift work enqueued by the actor is dispatched back to the main `Looper` via `LooperPoster`.

**Kotlin / Compose side:** `AppModelHolder` implements all four sink protocols (snapshot, command, searchQuery, isLoading). `BridgedSource<T>` wraps a JNI-bridged primitive as a Compose `MutableState` via `produceState` + listener, so Compose reads are transparently tracked.

The full data flow for a state change: Swift @Observable fires transaction → `Observations` loop wakes → JSON encode → JNI callback → Kotlin `mutableStateOf` update → Compose recomposition. Round trip: ~100–150µs.

### How Skip Fuse Works

Skip Fuse builds on the same Swift Android SDK. The difference is in what it automates and what it adds.

**Bridge automation:** When a module sets `bridging: true`, the skip-bridge build plugin scans Swift declarations and auto-generates JNI glue. No hand-written `@_cdecl` functions or sink protocols needed. Swift reference types are passed through JNI as opaque `Int64` pointers (`SwiftObjectPointer`) — the Kotlin side holds a numeric handle; ARC release is triggered when the Kotlin wrapper is GC'd. This avoids serializing Swift objects to JSON just to pass them across the boundary.

**@Observable → Compose integration:** SkipFuseUI maps SwiftUI property wrappers to Compose state. A `@Observable` class on the Swift side drives Compose recomposition without explicit "snapshot pump" code — the bridge generates the observation subscription automatically. From the developer's perspective, you write SwiftUI on both platforms and Compose renders it on Android.

**UI layer:** SkipFuseUI dispatches to SkipUI (a full Kotlin/Compose reimplementation of the SwiftUI API surface). All Compose views exposed by SkipUI have a `Compose(context:)` method so they can be embedded in a pure Compose tree, and vice versa with `ComposeView`. Mixed-mode apps can have some modules in Fuse (native Swift) and others in Lite (transpiled Kotlin).

## What observable-cross Does Well

### 1. Alignment with the Official Ecosystem

This project sits directly on Apple's Swift SDK for Android and swiftlang/swift-java. There is no commercial layer and no dependency on a third-party reimplementation of Foundation or SwiftUI. As swift-java matures (and it is moving fast), improvements land directly in this project's foundation. Contributing issues or patterns upstream has a clear path.

### 2. JavaUIActor + LooperExecutor

This is the project's most novel contribution. Pinning a Swift global actor to Android's main `Looper` via a custom SE-0392 serial executor is not covered by swift-java, Skip, or any published example. It solves a real problem — "where does Swift async run on Android, and how does it relate to Compose's thread model?" — with a principled solution. The `assumeIsolated` workaround (hand-rolled because stdlib only special-cases MainActor) is exactly the right engineering. This pattern deserves to be a published package.

### 3. Echo-Dedup Architecture for Two-Way Bindings

The trust-boundary dedup strategy — Kotlin writes dedupe on Kotlin side (`BridgedSource` skips echoes), Swift writes dedupe on Swift side (`AndroidBinding.lastSetterValue`) — is clean and correct. It avoids the classic round-trip loop in two-way bridged state without adding locks or complexity. Skip Fuse almost certainly solves this internally, but this project's solution is transparent and auditable.

### 4. Idiomatic Kotlin/Compose on Android

By not translating SwiftUI to Compose, Android UI can be fully idiomatic — every Compose feature, every Material 3 component, every platform-specific affordance is available without hitting an adapter boundary. For an app where Android UX quality matters, this is a genuine advantage over SkipUI's translation layer.

### 5. Strict Concurrency Throughout

SE-0461 actor isolation, region isolation (SE-0414), and near-zero `@unchecked Sendable` in core gives this project a rare property: the Swift compiler verifies concurrency correctness across the JNI boundary, not just within the Swift side. Skip Fuse's generated glue code necessarily includes unchecked annotations.

## What observable-cross Could Learn from Skip Fuse

### 1. SwiftObjectPointer — Pass Swift References, Not JSON

**The gap:** `AndroidSnapshot` encodes the entire `AppState` to JSON on every observation transaction. This is fast enough (~100µs for a typical snapshot) but it wastes work when only one property changed.

**Skip's solution:** `SwiftObjectPointer` passes an opaque `Int64` through JNI — the Kotlin side holds a handle to the live Swift object. Kotlin can call individual property getters via additional JNI thunks rather than deserializing a full JSON snapshot. The Swift runtime handles ARC retain/release when the Kotlin wrapper is constructed/GC'd.

**Adoption path:** swift-java JNI mode already supports generating property accessors (not just full-snapshot callbacks). The `AppState` class could be exposed as a set of individual `appcoreGetStories()`, `appcoreGetLastRefreshedAt()` etc. JNI getters, with a single "state changed" notification (no payload). Kotlin then pulls only the values it needs. This is particularly valuable for list-heavy state where JSON encoding is the bottleneck.

**Caveat:** The current JSON snapshot approach has the advantage of simplicity and debuggability — the wire format is human-readable. Keep JSON for commands and complex structured data; consider opaque pointer + property getters for frequently-updated scalars.

### 2. Build Plugin Automation for Sink Protocols

**The gap:** Adding a new bridged property today requires: (a) adding a `@_cdecl` JNI entry point, (b) defining a new sink protocol, (c) adding an `AndroidBinding` declaration in `Bridge.swift`, (d) implementing the sink in `AppModelHolder.kt`, and (e) adding a `BridgedSource` in Kotlin. Five touch points for one property.

**Skip's solution:** `bridging: true` in `skip.yml` generates all of this automatically. The developer annotates Swift declarations; the build plugin does the rest.

**Adoption path:** swift-java's `jextract` already handles step (a) automatically. Steps (b)–(e) are project-specific patterns that could be driven by a custom Gradle/SwiftPM plugin or by upstream contributions to swift-java. The `AndroidBinding` + `BridgedSource` pattern is generic enough to live in a small library package rather than being re-implemented per project.

### 3. Mixed-Mode Composition Pattern

**The gap:** AppCoreAndroid is a monolithic bridge module — one set of primitives covers all state. There's no facility for mixing different bridging strategies per module.

**Skip's insight:** Some modules benefit from full Fuse-style object passing; others benefit from the simpler Lite-style transpiled-to-Kotlin approach. Mixed-mode composition lets you pay only the costs relevant to each module.

**Adoption path:** Less directly applicable here since this project isn't using transpilation. But the composition principle applies: the `AndroidSnapshot` / `AndroidBinding` / `AndroidCommands` primitives are already designed to be composed. Making them a standalone `swift-android-observe` package (importable by other Swift+Android projects) would be the equivalent move.

### 4. Gradle Build Plugin Polish

**The gap:** The current Gradle build tasks (`buildSwiftAarch64`, `jextractAppCoreAndroid`) are functional but brittle — hardcoded arch, no incremental build support, no ABI split.

**Skip's solution:** Their Gradle plugin handles multi-arch builds, incremental `.so` packaging, and ABI splits out of the box. The build configuration is declarative.

**Adoption path:** This is low-urgency for a single-app project but matters if this becomes a reference implementation others follow. Adopting the `swift-java` Gradle plugin (which is being developed upstream) rather than custom `Exec` tasks would be the right long-term move.

## What to Keep, What to Change, What to Contribute

### Keep

- `@JavaUIActor` + `LooperExecutor` — novel, correct, and not available elsewhere
- `AndroidSnapshot` + `AndroidBinding` + echo-dedup — clean, explicit, auditable
- No UI sharing — Android UI in idiomatic Kotlin/Compose is the right call for quality
- Strict concurrency: SE-0461 isolation, no `nonisolated(unsafe)` in core
- JSON wire format at the JNI boundary — correct default for structured data

### Change

**Short-term:**
- Migrate from manual `Exec` Gradle tasks to swift-java's official Gradle plugin as it stabilizes. This reduces maintenance and gains multi-arch and incremental build support.
- Add ARM64 + x86-64 ABI splits. Currently only `arm64-v8a` is built; emulators and x86 devices need `x86_64`.
- Consider externalizing `AndroidSnapshot<S>`, `AndroidBinding<Root, Value>`, and `AndroidCommands<C>` into a standalone `swift-android-observe` Swift package. This makes the patterns reusable and gives the project an identity beyond "example app."

**Medium-term:**
- Evaluate the opaque-pointer approach (`SwiftObjectPointer`-style) for high-frequency scalar bindings (e.g., `isLoading`). If the snapshot round-trip becomes a bottleneck as state grows, individual property getters via JNI are a natural next step that swift-java already supports.
- Track swift-java's progress on protocol-as-callback interface support. The sink protocols (`SnapshotSink`, `CommandSink`, etc.) are already expressed as Swift protocols; as jextract's protocol support matures, the hand-written interface wrappers can be replaced with generated ones.

### Contribute Upstream

This project is doing things that don't exist anywhere in the official toolchain. The most valuable contributions:

1. **Publish `JavaUIActor` + `LooperExecutor` as a package** (e.g., `swift-android-actor`). This solves a problem every Swift-on-Android project will face: "how do I run Swift async on the Android main thread?" The SE-0392 custom executor approach, the `assumeIsolated` workaround, and the `Handler.post` round-trip are all non-obvious and well-solved here.

2. **Open an issue or pitch on swift-java** for native `@Observable` → Android integration. The `AndroidSnapshot` / `AndroidBinding` patterns could inform a future `swift-java` mode that auto-generates Compose-ready observation glue from `@Observable` Swift types. This would close the biggest gap between swift-java and Skip Fuse.

3. **Document the echo-dedup pattern** in a Swift Forums post or swift-java issue. Two-way bindings across a JNI boundary with correct round-trip suppression is a real problem; the current solution here is one of the cleanest public implementations.

## Direction Assessment

The project's stated goals — "integrate Swift @Observables into Jetpack Compose transparently" and "invoke Kotlin/Java API from Swift with zero setup" — are well-served by the current architecture. Both are largely achieved. The gaps that remain (boilerplate per property, single-arch builds, manual Gradle tasks) are engineering polish issues, not architectural problems.

Skip Fuse solves these gaps with a commercial product. This project solves them by being closer to the metal and feeding improvements back to the official tools. That is a harder path with a higher long-term payoff: every improvement here is a contribution to the official Swift-on-Android story, whereas every improvement to Skip Fuse stays proprietary.

**Should you adopt Skip Fuse?** Not as a dependency. The opaque-pointer technique and build automation ideas are worth borrowing; the SwiftUI-on-Android layer is not (you've correctly decided Android UI should be Kotlin/Compose). The commercial license introduces a dependency incompatible with the project's "demonstrate the official path" thesis.

**Should you contribute to swift-java and the Android SDK?** Yes. The two patterns most worth upstreaming — `JavaUIActor`/`LooperExecutor` and the `@Observable`→Compose observation bridge — don't exist anywhere in the official toolchain. Contributing them would accelerate the entire Swift-on-Android ecosystem and validate this project's approach as the reference implementation.
